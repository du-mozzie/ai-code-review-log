## 3. 其他问题清单

### 3.1. 架构与设计层面问题（严重性：一般）

1. **系统间依赖的耦合性高 (Siloed Integration)**
    * **描述：** 故障记录显示，系统间的集成（如金蝶、慧数通）严重依赖于外部的**权限状态**和**数据格式约定**。这种集成模式是高度耦合的。
    * **技术依据：** 违反了微服务中“服务自治性”和“数据隔离”的原则。一个系统的权限问题直接影响了另一个系统的核心业务。
    * **影响：** 系统的演进成本极高。如果金蝶的权限模型或数据结构发生变化，所有集成方都需要同步修改，形成**“集成黑盒”**。

2. **数据转换层（Mapping Layer）的脆弱性**
    * **描述：** 多个故障（如“bak”后缀问题）暴露了数据转换层对**外部系统细节的过度依赖**。
    * **技术依据：** 缺乏对外部API契约变化的抽象和解耦。
    * **影响：** 代码的可维护性差，任何外部系统的小改动都可能导致核心数据管道的崩溃。

### 3.2. 工程化与可观测性问题（严重性：可优化项）

1. **异常处理与日志粒度不足**
    * **描述：** 故障信息是“错误信息”的字符串描述，缺乏结构化的错误码、堆栈信息（如果存在）和关键上下文数据。
    * **技术依据：** 生产级系统应采用结构化日志（JSON/Logback），便于通过 ELK/Splunk 等工具进行高效的**故障根因分析 (RCA)**。
    * **影响：** 故障定位时间（MTTR）会非常长，因为需要大量人工解析非结构化文本。

2. **数据驱动的决策机制缺失**
    * **描述：** CSV 文件本身是故障记录，但它没有体现出**从这些故障中提炼出的、可执行的系统优化项**。
    * **技术依据：** 缺乏数据驱动的反馈闭环。
    * **影响：** 团队可能只是在“治标”（修复单个错误），而没有从**数据中学习（如：应该在哪个层级增加校验，应该增加哪个缓存）**，导致技术债持续积累。

## 4. 改进建议

### 4.1. 立即行动（修复致命/严重问题）

1. **强化跨系统权限治理：**
    * **建议：** 引入一个**中央权限服务（或策略引擎）**。所有调用外部系统（如金蝶）的请求，必须先通过该服务进行**动态的、基于上下文的权限校验**，而不是简单地依赖静态的授权令牌。
2. **引入数据适配器/反序列化策略（针对映射问题）：**
    * **建议：** 在数据转换层（Mapper）设计中，不要使用硬编码的字符串拼接。应使用**配置驱动（Configuration-driven）**或**基于字段属性的动态转换逻辑**。对于“需要自动加后缀”的场景，应在配置中明确定义这个规则，而不是在代码中硬编码一个特定的值。
3. **实现消息/API 的幂等性（针对重复传输）：**
    * **建议：** 对所有涉及状态变更或数据提交的 API 接口，强制要求调用方提供一个**业务级别的唯一标识（Idempotency Key）**。在处理前，检查该 Key 是否已处理，避免重复操作。

### 4.2. 长期架构演进（提升可维护性）

1. **构建结构化故障记录系统 (Structured Observability):**
    * **建议：** 将未来的故障记录从 CSV 迁移到**结构化的日志系统（如 Elasticsearch + Logstash + Kibana/Grafana）**。每条故障应包含：`timestamp`, `service_name`, `trace_id`, `error_type`, `error_code`, `request_payload_hash`, `stack_trace`。
2. **解耦外部集成层 (Anti-Corruption Layer - ACL):**
    * **建议：** 在面向金蝶、慧数通等外部系统的接口层，**强制实施“反腐蚀层”**。该层负责将外部系统的复杂、混乱的数据模型和权限模型，**翻译（Translate）成内部清晰、一致的领域模型**。这样，内部核心业务系统就不需要关心外部系统的任何细节变化。
3. **引入契约测试 (Contract Testing):**
    * **建议：** 对所有外部依赖的 API 接口，应建立**契约测试**。通过定义 OpenAPI/Swagger 规范，并在 CI/CD 流程中自动化测试**客户端（调用方）**和**服务端（被调用方）**之间的契约一致性，提前发现集成问题。
