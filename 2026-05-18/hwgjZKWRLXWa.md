## 3. 其他问题清单

### A. 架构设计与职责划分

1.  **日志存储架构缺陷：** 采用 CSV 文件作为主要的故障数据载体，这严重违反了生产级系统对**可观测性（Observability）**的要求。日志应通过结构化日志系统（如 Logstash/Fluentd $\rightarrow$ Elasticsearch/Prometheus）进行摄取，便于**实时聚合、指标化和趋势分析**。
2.  **错误信息粒度不一致：** 故障日志的字段（`错误信息` 字段）的内容质量参差不齐，既有具体的业务错误（如“解析字段异常”），也有高层级的环境问题（如“权限不足”）。这表明**异常捕获层（Exception Handling Layer）**在不同服务间的设计标准不一，导致下游的故障分析工具无法统一处理。

### B. 性能与资源使用

*   **CSV I/O 效率：** 虽然当前数据量可能不大，但如果此文件每日产生大量记录，使用 CSV 进行增量更新和读取，I/O 效率远低于使用更高效的二进制格式（如 Parquet）或专用的日志存储服务。

### C. 安全性

*   **数据敏感性：** 虽然此文件本身看起来不含敏感数据，但如果后续引入更详细的错误信息，需要确保所有涉及客户或系统配置信息的字段都经过严格的**数据脱敏处理**，以符合数据安全规范。

## 4. 改进建议

### 1. 强制结构化日志（最关键的架构改进）

**目标：** 彻底放弃依赖 CSV 文件作为主要故障记录方式。

*   **方案：** 引入**结构化日志标准（JSON/Protobuf）**。每一个故障事件都应该是一个独立的、带丰富上下文的 JSON 对象。
*   **示例结构（伪代码）：**

```json
{
  "timestamp": "2023-10-27T10:00:00.123Z",
  "traceId": "a1b2c3d4e5f6g7h8", // 必须添加
  "serviceName": "fault-qa-app",
  "level": "ERROR",
  "faultType": "PERMISSION_DENIED",
  "faultDetails": {
    "taskName": "旺店通销售出库单到金蝶销售出库单",
    "taskId": "19551",
    "errorCode": "20001",
    "errorMessage": "授权权限不足",
    "rawResponse": { /* 原始响应体 */ },
    "context": {
        "customerContext": "金蝶云星空_企业版",
        "customerAction": "单据创建"
    }
  },
  "stackTrace": "...", // 捕获完整的堆栈信息
  "suggestedAction": "联系客户重新授权应用令牌。"
}
```

### 2. 增强异常捕获与上下文注入

*   **在服务层：** 确保在捕获到任何业务异常时，**自动注入**以下上下文信息到日志中：
    *   **请求ID (Trace ID)：** 用于关联所有相关操作。
    *   **输入参数快照：** 对于参数校验失败的错误（如 `旺店通` 的例子），记录失败时的输入数据结构。
    *   **系统状态信息：** 记录服务所在的数据中心ID、配置版本等。

### 3. 规范化数据映射（集成层）

*   **解耦配置：** 将如“编码 `JSFS16` 对应 `JSFS04`”这类**业务映射规则**，从硬编码的 CSV/代码逻辑中剥离出来，放入**配置服务**或**元数据服务**中。这样，故障日志中只需记录“映射服务调用失败”，而不需要记录具体的映射规则细节。

### 4. 严格的 CSV 格式约束（如果必须保留 CSV）

*   **Schema 定义：** 如果必须使用 CSV，必须为每一列定义严格的、非模糊的 Schema。例如，`错误信息` 字段不应是自由文本，而应是 `ERROR_CODE` (字段A) + `DESCRIPTION` (字段B) 的组合，便于后续的 ETL 过程。
