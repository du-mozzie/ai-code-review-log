### 3. 其他问题清单

#### 3.1. 安全性与配置问题 (高风险)
1. **硬编码的外部服务地址**：
    * **问题**：`"http://121.36.71.64:1741/api/v1/code-review/exec-review/${{ secrets.ACCESS_TOKEN }}"` 存在硬编码的 IP 地址和端口。
    * **技术依据**：生产环境应使用配置中心（如 Vault, Kubernetes ConfigMap, 或 GitHub Secrets 结合环境配置）来管理外部服务的 Endpoint，而非直接写死在 Workflow 中。
    * **影响**：如果 Review Service 的地址发生变化，需要修改 Workflow 文件并重新部署，缺乏弹性。

2. **Token 的使用方式**：
    * **问题**：`ACCESS_TOKEN` 被直接用于 API 调用。
    * **技术依据**：应评估该 Token 的权限范围。如果 Token 权限过高，应考虑使用更精细的、基于角色的访问控制 (RBAC) 机制，而不是一个通用的 Access Token。

#### 3.2. 可维护性与工程化问题 (中风险)
1. **Shell 脚本的复杂性**：
    * **问题**：大量的 `echo` 和变量赋值操作分散在多个步骤中，导致流程的可读性降低。
    * **技术依据**：在复杂流程中，应将业务逻辑（如 Diff 生成、Payload 构建）封装到独立的脚本或函数中，提高流程的可测试性和可维护性。

2. **依赖安装的临时性**：
    * **问题**：`Install jq` 仅在当前 Job 中执行。
    * **技术依据**：如果后续步骤依赖 `jq`，最佳实践是在一个单独的 `setup-dependencies` 步骤中安装，或者使用更强大的基础镜像。

### 4. 改进建议

#### 4.1. 健壮性与可靠性改进 (强制执行)
1. **引入重试机制**：
    * **方案**：将 `Send to Review Service` 步骤封装在一个循环或使用 `actions/retry` 策略中，专门针对 HTTP 5xx 错误和网络超时进行重试（例如 3 次，每次间隔 10 秒）。

2. **配置外部依赖**：
    * **方案**：将 Review Service 的 URL 移至 GitHub Secrets 中，命名为 `REVIEW_SERVICE_URL`，而不是硬编码。

#### 4.2. 代码质量与可维护性改进
1. **抽象 Diff 生成逻辑**：
    * **方案**：创建一个单独的 Shell 脚本（例如 `generate_diff.sh`），将 `git diff` 的复杂逻辑封装进去，然后在 Workflow 中仅调用该脚本，使 Workflow 步骤更清晰。

2. **使用更安全的 JSON 构造方式 (可选)**：
    * **方案**：如果 Payload 结构变得复杂，考虑使用一个小型 Go 或 Python 脚本来生成 JSON，而不是依赖 `jq` 拼接复杂的 `--rawfile` 结构，以提高复杂数据结构构建的健壮性。

#### 4.3. 架构层面反思
* **职责分离（Separation of Concerns）**：当前 Workflow 文件承担了：环境信息获取、Git 操作（Diff）、数据转换（JSON 构造）、网络传输（Curl）的所有职责。在大型项目中，应将 **Git 交互层**、**数据模型层** 和 **外部服务适配层** 分离到不同的 Action 或脚本中。
* **系统边界**：该 CI 流程是典型的“触发 -> 数据准备 -> 外部系统调用”的模式。如果 Review Service 成为一个高频、高依赖的瓶颈，应考虑引入**异步队列**（如 Kafka/RabbitMQ），将 Code Review 请求放入队列，CI 仅负责提交请求，Review Service 异步处理，从而解耦 CI 流程的失败风险与外部服务的实时可用性。
