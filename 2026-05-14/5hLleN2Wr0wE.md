### 3. 其他问题清单

#### 3.1. 性能与资源使用 (一般问题)
*   **问题描述**：如上所述，最终的元数据归一化步骤存在性能瓶颈。
*   **技术依据**：多次迭代和集合操作的效率。在数据预处理管道中，追求 $O(N)$ 级别的复杂度是最佳实践。
*   **影响**：在处理数万甚至数十万个 Chunk 时，数据准备时间会线性增长，成为 I/O 或 CPU 瓶颈。

#### 3.2. 代码可读性与工程化 (可优化项)
*   **问题描述**：元数据处理逻辑过于复杂，混合了初始化、增量填充和最终归一化三个阶段，使得代码的意图难以快速把握。
*   **技术依据**：违反单一职责原则（SRP）。`DataPreparationModule` 承担了分割、基础标记、缺失值填充和最终规范化的职责。
*   **影响**：未来维护者需要花费更多时间理解这个“三步走”的元数据逻辑。

#### 3.3. 健壮性 (可优化项)
*   **问题描述**：当 `chunk.metadata` 中已存在某个键，但其值类型不符合预期（例如，期望字符串但意外是数字），代码没有进行类型校验或转换。
*   **技术依据**：运行时数据校验的缺失。
*   **影响**：虽然最终归一化会覆盖，但如果后续代码依赖于特定类型的字段，此处的隐式类型转换可能在运行时引发更深层次的错误。

### 4. 改进建议

#### 4.1. 优化元数据生成流程（核心改进）
**目标**：将流程重构为一次性构建 Schema 映射，然后一次性填充所有 Chunk。

**建议实现思路（伪代码/结构）：**

1.  **定义 Schema 映射配置**：将标题层级与所需的 Metadata Key 显式映射，而不是依赖硬编码的 `default_metadata`。

    ```python
    # 假设的配置结构
    HEADER_MAP = {
        "#": "header_1",
        "##": "header_2",
        "###": "header_3",
    }
    EXPECTED_KEYS = set(HEADER_MAP.values()) | {"chunk_id", "batch_index", "chunk_size"}
    ```

2.  **初始化与填充（单次遍历）**：遍历 `chunks`，对每个 Chunk 执行以下操作：
    *   确保 `chunk_id` 存在。
    *   设置 `batch_index` 和 `chunk_size`。
    *   **迭代** `HEADER_MAP`，根据当前的标题结构，计算并设置对应的 `header_X` 字段。

3.  **最终归一化（高效版本）**：在所有 Chunk 处理完毕后，可以**跳过**原来的耗时归一化循环。因为我们在步骤 2 中已经确保了所有 Chunk 都拥有了所有预期字段（即所有 `EXPECTED_KEYS`），所以不需要再进行昂贵的二次迭代查找缺失字段。

    *   **如果下游 Milvus 严格要求所有字段必须存在（即使为空）**：则在步骤 2 中，对所有预期键进行初始化（如 `chunk.metadata.setdefault(key, "")`），这样可以完全避免最后那个慢速的归一化步骤。

#### 4.2. 解耦标题配置
将 `headers_to_split_on` 的定义从代码中提取出来，作为配置参数或从外部配置文件加载，以提高系统的可配置性（OCP）。

#### 4.3. 增强日志与可观测性
在元数据生成阶段，记录关键的统计信息，例如：
*   成功分割的 Chunk 数量。
*   有多少 Chunk 缺少 `chunk_id` 并被生成了新 ID。

### 5. 架构层面反思

**结构性缺陷**：**强耦合于下游存储的 Schema 约束。**

该模块的设计哲学是**“为了满足下游特定批量写入的要求，在数据准备阶段强行构建一个统一的、规范化的 Schema 结构”**。

1.  **职责边界模糊**：数据准备模块承担了“语义解析”（Markdown结构）和“数据格式化”（Schema对齐）的双重责任。在更健壮的微服务架构中，应将**数据解析/结构化**与**数据规范化/适配**分离开来。
2.  **演进成本高**：如果未来更换向量数据库（例如，从 Milvus 切换到 Pinecone 或 Weaviate），或者数据结构要求改变，需要重写这个复杂的元数据统一逻辑。

**更优的系统设计或演进路径**：

*   **引入 Metadata Service/Schema Registry**：对于关键的元数据字段，可以考虑引入一个轻量级的 Schema 注册中心。数据准备模块只负责**解析**原始数据结构，然后查询 Schema Registry 获取**目标 Schema 模板**，最后再进行适配。
*   **数据管道模式 (Pipeline Pattern)**：将整个数据准备过程视为一个流水线（Pipeline）。
    *   Stage 1: Raw Chunking (生成内容)
    *   Stage 2: Enrichment (添加基础元数据：ID, Size)
    *   Stage 3: Schema Alignment (根据预设模板，补全所有必需的字段)
    *   Stage 4: Final Validation (检查所有字段的类型和存在性)

通过这种方式，可以确保每个阶段只关注一个核心职责，极大地提高了模块的解耦性、可测试性和未来系统的可扩展性。
