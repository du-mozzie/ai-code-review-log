### 3. 其他问题清单

#### 1. 架构层面反思

*   **结构性缺陷（职责划分）**: 设计清晰，但存在**“检索层”与“决策层/路由层”的耦合度较高**。`query-route.md` 明确提出了路由，但路由决策层（LLM Router）需要与下游的多个检索器（KB-1, KB-2, SQL等）紧密耦合。
    *   **反思**: 这种耦合是必要的，但需要确保路由决策层（LLM Router）的**输出结构（`RouteQuery` Pydantic 模型）**是高度稳定和版本化的，确保下游的路由映射（`ROUTES` 字典）的同步性。
*   **演进路径建议**: 您的设计路径是 **“轻量 $\rightarrow$ 复杂 $\rightarrow$ 架构级扩展”** 的完美演进路线。
    1.  **阶段一 (MVP)**: 优先实现**元数据过滤**（最快提升准确率）+ **查询改写**（提升初次召回）。
    2.  **阶段二 (增强)**: 引入**混合检索**（解决关键词遗漏）+ **规则/语义路由**（解决多源接入）。
    3.  **阶段三 (成熟)**: 引入**查询重构**（解决复杂意图）+ **多路并行检索**（解决多知识源对比）。
*   **Agent 化潜力**: `query-route.md` 中提出的“路由目标扩展到工具/Agent”是架构进化的终点。系统应向 **Agent 编排器**演进，从一个检索管道（Retriever Pipeline）升级为一个**工具调用管道（Tool Calling Pipeline）**。

#### 2. 生产规范与工程化建议

*   **可观测性（观测性是核心）**: 这是所有优化策略能否在生产中被信任的基石。
    *   **建议**: 必须为所有关键路径（路由决策、Query Rewriting、混合融合、最终检索）设置 **Tracing**（如 OpenTelemetry），并记录关键指标：**Query -> Route Decision -> Final Retrieved Docs -> Final Answer Quality Score**。
*   **评估体系的刚性**: 您提到“一定要离线评估”。在生产中，这需要固化为 CI/CD 的一个**测试阶段**。
    *   **建议**: 维护一个**Golden Test Set**（包含高难度、边界情况的查询），并将其作为回归测试的一部分。任何对 RAG 逻辑（如 RRF 参数 $\alpha$、路由阈值）的修改，必须保证该测试集的性能指标（Recall@K, MRR）不下降。
*   **Token/成本监控的实时性**: 既然提到了成本敏感性，**必须将 LLM 调用成本和延迟作为核心监控指标**。
    *   **建议**: 对 LLM 调用的 Token 计数和耗时进行实时监控，设置**预警阈值**，一旦某类查询的 LLM 调用成本超过预设上限，自动触发降级策略（如禁用 HyDE）。

### 4. 改进建议（可执行、可演进）

#### 1. 强制性工程改进（High Priority）

*   **RRF 优先**: 无论在任何混合检索实现中，请**默认采用 RRF 融合策略**，并确保其在所有组件中被优先选用，以最大化鲁棒性。
*   **向量库配置检查清单**: 制定一个检查清单，确保在部署前，所有用于过滤的字段都已正确配置了**标量索引**。
*   **延迟分级**: 在代码中，对不同的检索策略（如纯向量 vs. 混合 vs. 路由决策）设置明确的**延迟 SLA**。如果 LLM 路由或 HyDE 触发，应在代码层面设置严格的超时和降级逻辑。

#### 2. 优化实现细节（Medium Priority）

*   **Query Rewriting 缓存策略**: 对查询改写的结果应采用**基于查询哈希的缓存**。缓存 Key 应该包含原始 Query 的哈希、使用的小模型 ID 以及上下文（如果引入了历史记录），确保缓存的精确性。
*   **语义路由的动态迭代**: 引入一个**反馈循环**。当用户对一个路由决策表示不满意（例如，用户点击了 `fallback` 路径），系统应记录该 Query 和决策，并标记该决策的置信度为低，用于指导后续的**示例库迭代**。

#### 3. 架构演进（Long Term）

*   **工具调用抽象**: 将 `query-route.md` 中定义的路由目标扩展到更通用的 **Tool/Agent 注册表**。这样，系统不再是硬编码 `ROUTES` 字典，而是通过动态注册的 Agent 来执行任务，极大增强了系统的灵活性和可扩展性，真正实现架构的“系统改造”。
