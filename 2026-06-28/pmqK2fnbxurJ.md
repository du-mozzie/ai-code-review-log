### 3. 其他问题清单

1. **指标与迭代动作未建立度量映射**  
   *技术依据*：指标驱动开发（IDM）要求“评估动作”必须反哺“监控指标”。当前文档先列成本/延迟/质量指标，后接迭代建议，但未说明 20-50 条轨迹如何量化上述三维度。  
   *影响*：执行者知道“要迭代”，但不知道“迭代后如何证明效果提升”，导致实验数据无法沉淀为工程基线。

2. **缺乏自动化验证与防退化机制**  
   *技术依据*：LLM 系统具有概率性特征，Prompt/Schema 调整极易引发“局部最优但全局退化”（如降低幻觉率但导致延迟飙升）。生产级实践必须引入自动化回归测试集与阈值拦截。  
   *影响*：仅依赖人工 Review 20-50 条数据，无法覆盖长尾边界条件，高并发或极端输入下 SLA 易击穿。

3. **未明确工具 Schema 与 Prompt 的注册/发布规范**  
   *技术依据*：上下文工程中工具调用属于强类型契约。Schema 变更需遵循 OpenAPI/JSON Schema 规范，且需配合 Prompt 版本库进行灰度发布。  
   *影响*：多版本 Prompt 与工具 Schema 混用会导致运行时绑定失败，引发工具调用幻觉或 5xx 错误。

### 4. 改进建议

#### 4.1 文档工程化修正（立即执行）
将原行重构为标准技术段落，对齐上下文工程组件边界：
```markdown
### 迭代实践建议
建议优先构建 **20~50 条高覆盖真实任务轨迹** 作为最小可验证评测集（Golden Set）。基于该基线开展单变量迭代，核心关注以下组件：
- **检索器**：Top-K 召回率、相关性截断阈值
- **上下文压缩器**：信息保留比例 vs 输入 Token 消耗比
- **工具注册表**：Schema 变更兼容性、参数校验覆盖率
- **Prompt 模板**：结构化指令对齐度、Few-shot 样本有效性
所有迭代需配套自动化回归验证，确保核心指标（幻觉率、字段遗漏率、p95 延迟）不劣化基线。
```

#### 4.2 工程闭环落地方案
1. **轨迹数据结构标准化**  
   采用 JSONL 存储，关键字段：`task_id`, `input_context`, `expected_output`, `actual_output`, `metrics_scores`, `timestamp`, `prompt_version`。
2. **引入轻量级评估框架**  
   接入 `Ragas` / `DeepEval` / `Arize Phoenix` 实现自动化打分，输出结构化评测报告，替代人工抽检。
3. **Prompt/Schema 版本化治理**  
   使用 `DVC` 或 `Hashicorp Vault` 管理 Prompt 模板版本，变更需通过 `GitLab CI` 触发自动化回归流水线，阈值拦截后方可合并。

### 5. 架构层面反思

* **结构性缺陷**：当前内容停留在“经验建议”层面，未将其沉淀为**可编排的 AI 工程流水线**。上下文工程本质是“数据+提示词+模型”的概率系统，传统代码的确定性测试模型在此失效，必须引入**评估即代码（Eval-as-Code）** 与**实验追踪（Experiment Tracking）** 架构。
* **更优演进路径**：
  1. **基线化**：将 20-50 条轨迹抽象为 `ai-eval-baseline` 独立模块，与业务代码解耦，支持按环境（Dev/Staging/Prod）加载。
  2. **契约化**：工具 Schema 纳入 OpenAPI 注册中心，Prompt 模板纳入模板引擎（如 Mustache/Chug），实现运行时动态注入与版本回滚。
  3. **可观测性闭环**：在评测集产出后，自动映射至日志/链路追踪系统（OpenTelemetry），实现“评测失败 -> 告警 -> 基线降级 -> 人工介入”的 SLA 保障机制。

> **结论**：该 diff 具有正确的工程直觉，但缺乏生产级文档规范与实验可观测性设计。建议按上述方案补全结构、术语与闭环机制后提交。AI 系统迭代不是“调参游戏”，而是“数据驱动的工程实验”，文档应直接反映这一工程属性。
