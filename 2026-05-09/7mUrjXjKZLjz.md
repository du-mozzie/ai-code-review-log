### 3. 其他问题清单

#### 1. 架构设计与职责划分（严重问题）
*   **问题描述：** 当前的 `SelfQueryRetriever` 承担了**语义解析 (Intent Recognition)**、**结构化转换 (Transformation)** 和**最终查询生成 (Query Generation)** 三项职责。
*   **技术依据与影响：** 这种耦合使得系统对自然语言的理解能力边界清晰度很低。当新的查询类型（如集合运算、时间窗口查询）出现时，需要修改核心转换逻辑，这违反了高内聚、低耦合的原则。
*   **建议：** 拆分层级。推荐引入一个 **Query Intent Router**。该路由器首先判断用户意图是 $\text{Filter} \rightarrow \text{Router A}$ 还是 $\text{Sort/MinMax} \rightarrow \text{Router B}$，再将请求转发给对应的专业转换器。

#### 2. 性能与资源使用（可优化项）
*   **问题描述：** 每次查询都可能触发一次完整的 LLM 调用来生成查询结构，即使是简单的过滤查询，也存在不必要的延迟开销。
*   **技术依据与影响：** LLM 推理是昂贵的。对于高频的、简单的过滤查询，应避免过度依赖 LLM 来完成所有任务。
*   **建议：** 对于**高频、低复杂度的固定过滤查询**（例如，用户总是问“时长大于X”），可以考虑**预先注册**这些模板，使用**规则引擎 (Rule Engine)** 直接生成结构化查询，跳过 LLM 步骤，只保留 LLM 处理排序和复杂语义。

#### 3. 可观测性与异常处理（一般问题）
*   **问题描述：** 关键日志仅显示了 `Generated Query: query=' ' filter=None limit=1`。缺乏对**LLM生成过程的细粒度监控**。
*   **技术依据与影响：** 当查询失败时，仅凭最终结果判断困难。无法追溯是 LLM 输出了错误的结构，还是转换器解析失败。
*   **建议：** 在日志中捕获并记录**LLM的原始输出 (Raw LLM Output)** 和**意图识别结果**，以便在调试阶段能快速定位问题是出在**Prompt 设计**还是**解析逻辑**。

### 4. 改进建议

**A. 核心架构重构：引入查询意图驱动的策略模式**

**目标：** 彻底解耦过滤逻辑和排序/极值逻辑。

1.  **定义查询意图枚举 (Enum):**
    ```python
    class QueryIntent(Enum):
        FILTER = "FILTER"
        SORT_ASC = "SORT_ASC"
        MIN_VAL = "MIN_VAL"
        # ... 其他意图
    ```

2.  **设计转换器接口 (Interface):**
    ```python
    class QueryConverter(ABC):
        @abstractmethod
        def convert(self, natural_language: str) -> dict:
            """将自然语言转换为 {query, filter, sort, limit} 结构体"""
            pass
    ```

3.  **实现具体转换器 (Implementations):**
    *   `FilterConverter`: 负责将自然语言映射到 `{query, filter, limit}`。
    *   `SortConverter`: 负责将自然语言映射到 `{sort_by, order, limit}`。
    *   `MinMaxConverter`: 负责处理“最短/最长/最高/最低”等词汇，生成 `{sort_by, order, limit}`。

4.  **Router/Dispatcher:**
    在 `SelfQueryRetriever` 中，首先使用一个轻量级的模型（或基于关键词匹配的规则）判断用户查询属于哪一类意图，然后调用对应的 `QueryConverter` 来执行转换，最后将统一的结构体传递给 Milvus 客户端。

**B. 提升 LLM Prompt 的鲁棒性 (针对操作符)**

在 Prompt 中，除了告知 LLM 字段名和类型外，**明确提供一个操作符的字典**，强制其从预设的集合中选择，而不是自由发挥。

*   **示例 Prompt 改进：** “你可以使用的过滤操作符集合包括：`gt(field, value)` (大于), `lt(field, value)` (小于), `eq(field, value)` (等于)。请严格使用这些格式。”

**C. 针对“时间最短”的直接处理 (快速修复)**

对于您分析中明确指出的“时间最短的视频”这种明确的极值查询，可以设计一个**前置校验层**：

*   **规则：** 如果输入包含“最短”、“最长”、“最高”、“最低”，**忽略 LLM 的转换**，直接硬编码转换为对应的 Milvus 排序参数（`sort_by='length', order='ASC', limit=1`）。
*   **目的：** 确保对**高风险/高确定性**的查询类型，系统能以 100% 的准确性服务，绕过不稳定的 LLM 语义转换。
